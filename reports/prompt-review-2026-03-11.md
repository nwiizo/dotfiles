- **Git/GitHubワークフロー**: 「ブランチを切って作業してほしい」「コミットログとIssueを紐づけてCloseして」「v0.0.3としてリリースして」など、ブランチ戦略・Issue連携・セマンティックバージョニングを日常的に運用。66件のコミット関連指示があり、リリース管理まで一貫して実施している（Claude Code / 全プロジェクト）
- **Claude Codeエコシステム**: CLAUDE.md最適化、rules/skills/agents/commandsの分離設計（「home-から始めるようにしてほしい」）、memory管理、マルチエージェントオーケストレーション（ccswarm）など、Claude Codeの高度な機能を熟知。「Analyze CLAUDE.md and propose an extraction plan」というメタプロンプトを複数プロジェクトで再利用している（Claude Code / nwiizo, oitoriaezu-owarasero, secure-apis-ja）
- **書籍編集・文体制御**: 「である。だ。をです。ます。に変更」「先輩が後輩に向かって語るぐらいの敬語」「元の断定の力強さが失われた」「〇〇として構文っぽくなるのをそもそも防ぎたい」「それぞれが完璧に揃っているのもちょっと不愉快」など、文体のニュアンスを精緻に言語化。AIの出力に対する「不自然さ」を的確に指摘できている（Claude Code / oitoriaezu-owarasero, blogs, books）
- **プレゼンテーション設計**: Marpスライドの構成設計から視覚的な調整（「黒背景に青黒い文字なので全然見えません」「ORM が太字になっていない」）まで、発表資料の品質管理を主体的に実施している（Claude Code / 3shake-marp-templates, slides）

## 4. プロンプティング力の評価

### 強み

- **明確な対象読者・品質基準の指定**: 「初学者向けに丁寧に説明してほしい」「セキュリティエンジニア以外に向けてなぜ攻撃に繋がるのか」「先輩が後輩に向かって語るぐらいの敬語」のように、成果物の受け手と品質水準を具体的に示す能力が高い
- **参照コンテキストの積極的活用**: 82件のメッセージでURLを共有し、「../uriapa/ を参考にブランチを切って」「https://github.com/nrslib/takt を参考にしながら」「/Users/nwiizo/.../blogs のコマンド群を参考にレビューして」と、AIに必要な文脈を外部リソースや既存プロジェクトから与えている
- **段階的な作業管理**: 「TODOを作って一章ずつお願いします」「計画を立ててTODOを作成してほしい」「Highからお願いします」と、大きなタスクを管理可能な単位に分割し、優先度を指示する習慣がある（32件のTODO関連指示）
- **メタ知識の抽象化要求**: 「もっと深堀りして抽象的な学びを得てほしい」「具体的なフィードバックから抽象的な学びを得ることが大切」「小川哲 の視点を抽象化してコマンドにしてほしい」と、単なる作業ではなく知識の体系化を求めるプロンプティングが特徴的
- **AIの出力品質への鋭い感性**: 「以下が少しAIっぽいので表現を修正してほしい」「〇〇として構文っぽくなるのをそもそも防ぎたい」「それぞれが完璧に揃っているのもちょっと不愉快」と、AI生成テキストの不自然さを的確に言語化できている

### 改善ポイント

- **「深堀り」ループの反復**: 「もう少し深堀りしてほしい」が68回出現（特にblogs: 33回）。深堀りの方向性が不明確なため、AIが汎用的な拡張を行い、再度「深堀り」を要求するサイクルが発生している → **改善提案**: 「〜の観点から深堀り」「具体的には〜について3倍の分量で」のように、深堀りの軸と到達点を指定する
- **「ちゃんと」の多用**: 64回出現。「ちゃんと画像を確認してから正しいか確認して」「ちゃんと前回のバージョンからの差分を出して」「ちゃんと調べて各コマンドを更新して」など、具体的な品質基準を省略して「ちゃんと」で代替する傾向 → **改善提案**: 「ちゃんと」を「全ファイルを対象に」「公式ドキュメントと照合して」「diffを出力して確認してから」のような具体的な手順に置き換える
- **文脈なし短文指示**: 「修正してほしいです。」（289件中、文脈を欠く短文が多い）のように、何をどう修正するかの文脈が直前の会話に依存しすぎるケースがある → **改善提案**: 「[ファイル名]の[箇所]を[方向性]に修正して」と1メッセージで完結する指示にする
- **要件の後出し**: 書籍の文体変換で「私はそう信じたい。というような自分に語る時はそのままで良い」「やはり、ビジネス書です」、スライドで「青い文字も見えづらいのでオレンジか赤か白に」と、初回指示に含めるべき制約を後から追加するパターンが複数プロジェクトで観察 → **改善提案**: 頻出タスク（文体変換、スライドデザイン、脆弱性診断）については制約チェックリストを.claude/commands/に用意する
- **セキュリティ情報のIssue記載**: 「平文のままがどのユーザーかも記載してほしい」→「形跡を跡形もなく消し去ってほしい」と、セキュリティ上の情報を公開リポジトリに残そうとして後から削除するパターン → **改善提案**: セキュリティ調査結果はローカルファイルに記録し、Issue/PRには対策のみを記載するルールを策定する

### 特徴的な癖

- **「〜してほしいです」文体**: 1,695件中、大半がこの文体。急ぎや不満時は「〜して下さい」「〜してほしい」（丁寧語省略）にトーンが変わる
- **プロジェクト横断的な参照**: 「../uriapa/ を参考に」「/Users/nwiizo/.../blogs のコマンド群を参考に」と、自分の過去の成果物を積極的にテンプレートとして再利用する
- **「抽象化」志向**: 書評から学びを抽出→コマンド化、PRレビューから学びを抽出→rules化、文学的観点の抽象化→エージェント化、と知識を再利用可能な形に変換する強い志向がある
- **即座の方向転換**: AIの出力が期待と異なる場合、「あまり修正が気に入らないかもです。元に戻してPLANをやって」「SREとしてみたいな構文は不要です。全体として削除して」と、ためらいなく方向転換する

## 5. AI活用スタイル

### 主体的に活用できている領域

- **書籍・文章のディレクション**: 文体・構成・対象読者の方針を自ら決定し、AIに実行を委ねている。修正が期待と異なる場合は即座に方向転換する。「最高の書籍を作るエージェント」「好奇心と自分だけの観点だけがAI時代に必要な能力になる。みたいな主張をして」と、コンテンツの方向性は完全に自分で把握している
- **セキュリティ診断の設計**: 脆弱性診断のフレームワーク（ブランチ命名規則、診断手順、レポート形式）を自ら設計し、それを複数プロジェクト（uriapa→itotoushi→rocky）に横展開している。「ホワイトハッカーとブラックハッカーを作成してシュミレーション」というエージェント活用の発想も主体的
- **Claude Code設定体系の設計**: CLAUDE.md最適化の指示テンプレート（「Analyze CLAUDE.md and propose an extraction plan...」）を自ら開発し、5プロジェクト以上で再利用。skills/commands/agents/rulesの分離設計を主導している
- **プレゼンテーション資料の品質管理**: スライドの視覚的品質（色、フォント、レイアウト）について具体的なフィードバックを行い、AIの出力を段階的に改善している

### AIへの依存が高い領域

- **設定ファイルのデバッグ**: Neovimの自動リロード問題やGhosttyの設定エラーなど、設定レベルの問題解決をAIに委ねている → **自立に向けた提案**: `:checkhealth`やghostty +validate-config等の診断コマンドを先に実行し、結果をAIに共有する習慣をつける
- **本番環境の操作**: 「ssh でログインして実際の実装を確認して」「本番環境で適切にハッシュ化して」とAIに本番操作を依頼している → **自立に向けた提案**: 本番操作は手順書をAIに生成させ、実行は自分で行うフローを確立する
- **技術選定の詳細**: 「より最新で新しい方の技術選定をしてほしい」「ジョブ理論を考えて必要な機能を整理して」と、技術選定の具体的な判断をAIに委ねるケースがある → **自立に向けた提案**: 判断基準（パフォーマンス/メンテナンス性/エコシステム成熟度等）を先に提示し、AIにはそれに基づく比較表を生成させる

### ツール別の活用傾向

- **Claude Code（唯一の検出ツール）**: ソフトウェア開発ライフサイクルの全域で活用。特に以下の特徴的な使い方が観察される:
  - **`--dangerously-skip-permissions`** での自律実行モードを常用（`c` abbreviation）
  - **マルチエージェント**: ccswarmでBackend/Frontend/DevOps/QAエージェントを同時起動（約100セッション）
  - **スキル/コマンドの自己拡張**: `/full-review`, `/nwiizo-style-review` 等のカスタムコマンドを開発・活用
  - **CLAUDE.md最適化テンプレート**: 「Analyze CLAUDE.md and propose an extraction plan...」を複数プロジェクトで流用
  - **週あたり300-560件**の高頻度利用（5週間の平均: 385件/週）

### 推奨される次のステップ

1. **頻出タスクのテンプレート化**: 文体変換・スライド作成・脆弱性診断・ブログ執筆について、制約条件を網羅したcommandテンプレートを`.claude/commands/`に作成する。これにより「要件の後出し」と「ちゃんと依存」の両方を解消できる
2. **「深堀り」の構造化**: 「深堀りしてほしい」を使う前に、深堀りの軸（技術的詳細/ビジネス影響/歴史的文脈/対比分析）と到達点（文字数/具体例の数/引用の追加）を明示する習慣をつける。例: 「Leptos のシグナルシステムについて、ReactのuseStateとの比較を交えて2000字で深堀りして」
3. **セキュリティ情報取り扱いルールの策定**: 脆弱性診断結果やパスワード情報はローカルファイルに記録し、Issue/PRには対策のみを記載するルールを`.claude/rules/security.md`に追加する
4. **長期間分析の定期実行**: 本レポートは60日分のデータに基づいているが、四半期ごとに`/prompt-review 90`を実行し、成長の軌跡を継続的に追跡することを推奨する
